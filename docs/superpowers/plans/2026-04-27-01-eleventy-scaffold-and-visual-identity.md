# Plan 01 — Eleventy Scaffold and Visual Identity Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the hand-authored static HTML site with an Eleventy build, and apply the "field journal" visual identity to a single reference page (Morecambe project) so the look is locked before we port the rest of the content in Plan 02.

**Architecture:** Eleventy 3.x (ESM config) generates a static site into `_site/`. Sources live in `src/`. Sidebar + main-panel layout, monochrome palette, EB Garamond (display) + Inter (body) via Google Fonts as the licensed-font fallback. The existing `assets/media/` tree, `nav.html`, `aboutme.html`, `contact.html`, `index.html`, `photos/portfolio.html`, `projects/*.html`, `blog/*.html`, and the `scripts/`/`tests/` directories remain untouched on disk during this plan — they'll be removed in Plan 02 once their content is ported. GitHub Actions builds and deploys to GitHub Pages.

**Tech Stack:** Eleventy 3, Nunjucks templates, vanilla CSS (no preprocessor), Vitest (lightweight, ESM-native — replaces existing Jest setup), GitHub Actions for deploy.

---

## Pre-flight notes for the implementing agent

- The repo currently uses CRLF line endings on Windows. Don't fight it; `.gitattributes` is fine to leave as-is.
- The existing `package.json` has Jest + Husky. We're swapping Jest for Vitest because Eleventy 3 is ESM-only and the existing Jest config doesn't handle that cleanly; Husky stays.
- Existing `assets/media/` is referenced by the old gallery system. Don't move or rename anything in there during this plan — the old site still needs to render until Plan 02 cuts over.
- All Eleventy work happens under `src/` (new directory) and `.eleventy.js` at the root. The old files coexist.
- During this plan the dev server serves the new Eleventy site. The old static HTML is still in the repo but isn't built into `_site/`; opening the old files directly via `file://` or `python -m http.server` is the way to compare.
- **One reference page only in this plan: Morecambe.** Do NOT port other projects, About, Contact, the homepage proper, or anything else. Plan 02 does that.
- Test code uses `execFileSync` with argv arrays (never `execSync` with shell strings) — there is no user input flowing into commands but the safer primitive is the default in this repo.

## Task 4: Decide and document the font situation, then write head-meta partial

**Files:**
- Create: `docs/superpowers/fonts-decision.md`
- Create: `src/_includes/partials/head-meta.njk`

- [ ] **Step 1: Make the font decision and record it**

The spec says GT Sectra + Söhne are commercial. For this plan, default to the Google Fonts fallback pairing because it's free and doesn't block the visual review:

- Display: **EB Garamond** (Google Fonts) — closest free analogue to GT Sectra's editorial weight.
- Body: **Inter** (Google Fonts) — strong sans, well-tuned hinting, the closest free analogue to Söhne for body copy.
- Mono: system stack (`ui-monospace, SFMono-Regular, Menlo, monospace`).

Create `docs/superpowers/fonts-decision.md`:

```markdown
# Font decision (Plan 01)

**Decision:** Use Google Fonts pairing (EB Garamond + Inter) for the reference build.

**Rationale:** GT Sectra and Söhne are commercial; licensing isn't resolved at the time of Plan 01. The fallbacks are explicitly named in the spec. The Google Fonts pairing is good enough to lock in the visual direction; if the look is right with these fonts, it'll be at least as right with the commercial ones.

**Reversal path:** When/if GT Sectra and Söhne are licensed, replace the `<link>` in `head-meta.njk` and the `font-family` declarations in `site.css`. No structural changes needed.
```

```javascript
// Minimal sidebar toggle. The old client-side nav loader is gone;
// Eleventy renders the sidebar at build time.
(() => {
  const toggle = document.getElementById('sidebarToggle');
  const sidebar = document.querySelector('.sidebar');
  const overlay = document.getElementById('overlay');
  if (!toggle || !sidebar || !overlay) return;

  const open = () => {
    sidebar.classList.add('open');
    overlay.hidden = false;
    toggle.setAttribute('aria-expanded', 'true');
  };
  const close = () => {
    sidebar.classList.remove('open');
    overlay.hidden = true;
    toggle.setAttribute('aria-expanded', 'false');
  };

  toggle.addEventListener('click', () => {
    sidebar.classList.contains('open') ? close() : open();
  });
  overlay.addEventListener('click', close);
  window.addEventListener('keydown', (e) => { if (e.key === 'Escape') close(); });
})();
```

- [ ] **Step 1: Write the workflow**

Create `.github/workflows/deploy.yml`:

```yaml
name: Build and deploy

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test
      - run: npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: _site

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```
